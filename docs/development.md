# Development

## Prerequisites

- Node.js (any recent LTS) and npm.
- No browser-side tooling required тАФ the plugin is vanilla JS transpiled by Babel and packaged by the `tiddlywiki` CLI.

```bash
npm install
```

Installs `@babel/*`, `cross-env`, `diff-match-patch` (used by the Jest test shim only тАФ the runtime uses TW's bundled copy), `jest`, and `tiddlywiki`.

## Build pipeline

Babel transpiles ES modules in `src/` to CommonJS in `lib/` (required because TW's module loader is synchronous CommonJS). The `tiddlywiki.files` manifest then maps each `lib/*.js` file to its TW module title.

```bash
npm run babel           # src/ тЖТ lib/
npm run build-plugin    # babel + package plugin as build/timelord.tid
npm run seed-demo       # regenerate demo tiddlers from scripts/demo-content.js
npm run build-demo      # build the demo wiki HTML to build/index.html
npm run build-debug     # build the debug wiki HTML to build/index.html
npm run build-all       # build-plugin + seed-demo + build-demo
```

The plugin `.tid` file produced by `build-plugin` is drag-and-droppable into any TiddlyWiki тЙе 5.3.0.

## Development workflow

### Two wikis

- `demo/` тАФ canonical. Committed. `build-demo` always reads here. Revision tiddlers are gitignored (generated by `seed-demo`); content tiddlers are committed.
- `debug/` тАФ gitignored sandbox. `serve` / `build-debug` read here. Seeded from `demo/` on first use.

```bash
npm run serve          # transpile + dev server on localhost:8081 against debug/
npm run clean-serve    # seed demo data + wipe debug/ + re-seed from demo/ + serve
npm run clean-debug    # wipe debug/ and re-seed from demo/
```

### Demo content

Demo tiddler content and edit sequences are defined in `scripts/demo-content.js`. The seed engine (`scripts/seed-demo.js`) replays these through the Revisor class using the test mock runtime to produce valid `.tid` files with proper revision chains.

To add a new demo tiddler, add an entry to `demo-content.js` with a `name` and `versions` array. Each version specifies an action (`save`, `rename`, or `delete`), a `wait` (minutes to advance the clock), fields, and an optional `summary`.

To promote a sandbox change into the demo: edit `scripts/demo-content.js` and re-run `npm run seed-demo`.

### Typical edit cycle

1. Edit `src/revisor.js` / `src/listener.js` / `src/filters.js`.
2. `npm test` тАФ fast, no TW needed.
3. `npm run serve` and exercise the change in a real wiki.
4. When happy, `npm run build-plugin` to sanity-check the packaged output.

`npm run serve` runs `babel` as part of its pre-step, so source edits are picked up after a page reload. If you're editing a `.tid` UI tiddler, changes to the plugin folder require a rebuild тАФ easiest is Ctrl-C and re-run `npm run serve`.

## Testing

Jest with `babel-jest`. Tests run entirely on Node, no browser, no live TW.

```bash
npm test
npm run test:coverage
```

Structure under `tests/`:

| File | Purpose |
|------|---------|
| `mock-tw.js` | Minimal `$tw` stand-in: in-memory wiki store, `Tiddler` constructor, tag index, hooks registry, root widget event listener. |
| `dmp-shim.js` | Maps TW's internal `require("$:/core/modules/utils/diff-match-patch/...")` path to the npm `diff-match-patch` package, so real patch/apply cycles run in tests. |
| `setup.js` | Wires `mock-tw.js` and the shim into `global.$tw` before each test file. |
| `revisor.test.js` | Unit + integration tests for the `Revisor` class. Exercises every storage mode and all dedup paths. |
| `listener.test.js` | Hook registration and message routing. Covers every short-circuit guard. |
| `filters.test.js` | All three filter operators, including fallbacks. |
| `import.test.js` | Import behavior and edge cases. |

Coverage targets: ~90% statements/lines on core code. Do not drop below.

### Writing new tests

The mock `$tw` is intentionally minimal тАФ add methods to it only when a test needs them. Keep the mock side effectтАУfree between tests by resetting in `beforeEach` (see existing test files).

## Coding conventions

- All ES-module source lives under `src/`. Do **not** edit `lib/` directly тАФ it is a Babel output.
- No comments that restate the code. The top of `revisor.js` is the canonical place to document revision tiddler structure; mirror any field addition there **and** in [data-model.md](data-model.md).
- `AUTO_FIELDS` is exported from `revisor.js` and imported by `filters.js`. `listener.js` inlines an equivalent set for its own `tiddlerFieldsChanged` check тАФ keep it aligned with the exported list.
- Prefer pure functions on `Revisor`. UI-driven state (e.g. toggling the enabled flag around a restore) stays inside the method; hooks in `listener.js` stay thin.
- When introducing a new revision-level field, bump `SCHEMA_VERSION` in `revisor.js` and document the change in [schema-versioning.md](schema-versioning.md).

## Extending the plugin

### Adding a new filter operator

1. Export a function from `src/filters.js` matching TW's filter-operator signature `(source, operator, options) тЖТ string[]`.
2. Add a pointer to `tiddlywiki.files` only if you split operators into new files тАФ otherwise no change needed; the single `filters.js` module already exposes each export as an operator.
3. Add tests in `tests/filters.test.js`.

### Adding a new revision-level field

1. Document it in the header comment of `revisor.js` *and* in [data-model.md](data-model.md).
2. Write it in `addToHistory`. Strip it in `restoreFromRevision` if it should not appear on the live tiddler.
3. Bump `SCHEMA_VERSION`.
4. Ensure older revisions (missing the field) still reconstruct correctly тАФ either by making readers tolerant, or by keying on `getRevisionVersion(revision)`.

### Adding a new UI surface

`Revisions.tid` shows how to consume the filter operators. Prefer building on the existing operators rather than reaching into `revision-data` JSON directly from wikitext тАФ the operators hide `full`/`diff`/`delta` resolution, which wikitext cannot do.

## Debug logging

`Revisor` calls `console.log` on add / rename / capture / restore. `reconstructText` and `reconstructAllFields` call `console.warn` on patch-apply failures or unexpected chain state. These are visible in the browser devtools when running `npm run serve`.

## Continuous integration

[`.github/workflows/ci.yml`](../.github/workflows/ci.yml) runs on every push to `main` and every pull request. It installs dependencies with `npm ci`, runs `npm test`, then `npm run build-all`, and uploads `build/timelord.tid` as a workflow artifact. PRs that break tests or the build are blocked from merging.

## Release

Releases are cut by pushing a `v<version>` tag. [`.github/workflows/release.yml`](../.github/workflows/release.yml) then runs tests, builds the plugin, and creates a GitHub Release with `build/timelord.tid` attached and auto-generated notes from the commit log.

### Version fields

| Field | Where | Purpose |
| ----- | ----- | ------- |
| `plugin.info`.`version` | [plugins/mblackman/timelord/plugin.info](../plugins/mblackman/timelord/plugin.info) | The plugin version users see in TiddlyWiki's plugin manager. **This is the field the release tag must match.** |
| `SCHEMA_VERSION` | [plugins/mblackman/timelord/src/revisor.js](../plugins/mblackman/timelord/src/revisor.js) | On-disk revision format version. Bump only on breaking data changes тАФ see [schema-versioning.md](schema-versioning.md). |
| `package.json`.`version` | repo root | Unused at runtime. Keep it in sync with `plugin.info`.`version` if you update it at all; the release workflow ignores it. |

The plugin version is independent of `SCHEMA_VERSION`. Most releases bump `plugin.info`.`version` only. Bump `SCHEMA_VERSION` **in addition** to the plugin version when the change meets the criteria in [schema-versioning.md](schema-versioning.md).

### Cutting a release

1. Ensure `main` is green on CI.
2. Decide the new version (semver тАФ breaking data format = major, new user-visible features = minor, bug fixes = patch).
3. Bump `plugin.info`.`version`. If the change affects revision tiddler format, also bump `SCHEMA_VERSION` and add a row to the history table in [schema-versioning.md](schema-versioning.md).
4. Update [CHANGELOG.md](../CHANGELOG.md): add a new version section at the top. List user-visible changes only тАФ developer-facing refactors belong in the commit log, not the changelog.
5. Commit with a message like `Release v<version>`.
6. Tag: `git tag v<version>` then `git push origin v<version>`. The tag version must match `plugin.info`.`version` exactly (no `v` prefix in `plugin.info`); the release workflow fails fast if they disagree.
7. Watch the Release workflow complete. The GitHub Release page will have `timelord.tid` attached and auto-generated notes from the commit log.

### Hotfixes

For an urgent fix on top of a shipped release, branch from the tag, apply the fix, bump the patch version, and follow the same tag flow. CI runs the same checks on the release branch.

### Manual build (only if CI is unavailable)

1. `npm test` тАФ must pass.
2. `npm run build-plugin`.
3. Upload `build/timelord.tid` to a manually-created GitHub Release.
